# v0.1 Reading Notes — Bus, Loop, FakeProvider, Add Tool

> Notes taken while reading the nanobot originals before writing PepperMill v0.1.
> Sources: `../../../nanobot/nanobot/{bus,agent,providers,channels}/...`

## bus/events.py

**What it does:** Defines two pure-data message types that flow between channels
and the agent: `InboundMessage` (channel → agent) and `OutboundMessage`
(agent → channel). They are `@dataclass`es — no behaviour, just shape.

**Inputs/outputs:** None. These are values, not actors.

**Key design choices:**
- `field(default_factory=list)` / `field(default_factory=dict)` — required for
  any mutable default in a dataclass. Sharing one list across all instances
  is the classic Python footgun this guards against.
- `session_key` is a property derived from `channel:chat_id`, with an override
  field for cases where a single chat needs multiple logical sessions.

**Python concepts new to me / worth practicing:**
- `from __future__ import annotations` — lazy evaluation of annotations, so we
  can use `list[str]` / `int | None` on Python 3.10+ without runtime cost.
- `@dataclass(frozen=True)` was an option — nanobot didn't use it because
  some fields are mutated. I'll keep the same choice.

## bus/queue.py

**What it does:** Wraps two `asyncio.Queue`s — `inbound` and `outbound`. The
publish/consume methods are thin async wrappers around `put`/`get`.

**Why so simple?** This is the deliberate decoupling point. Channels and the
agent never reference each other; they only know the bus. Adding logging,
metrics, or backpressure later means changing one file.

**Key design choices:**
- No `maxsize` — accepts arbitrary backlog. For v0.1, fine. v0.10 adds
  bounded pending queues for backpressure.

**Python concepts:**
- `asyncio.Queue` is the canonical async producer-consumer primitive.
- `Queue[T]` syntax (generic alias) gives the type checker what it needs to
  enforce element types.

## agent/loop.py (skim only)

**What it does (in 1 line):** Drives a state machine per inbound message; the
real LLM call happens inside `AgentRunner` which `loop._state_run` calls.

**For v0.1:** We're collapsing this to a single linear method (`run_once`).
The FSM is v0.3's concern.

---

## providers/base.py

**Key shape we will mirror in v0.1:**
- `LLMResponse`: `content`, `tool_calls`, `finish_reason`, plus a
  `has_tool_calls` property.
- `ToolCallRequest`: `id`, `name`, `arguments` (a dict).
- `should_execute_tools` property guards against gateway-injected tool calls
  under refusal/content_filter — v0.1 skips this, the FakeProvider always
  produces well-formed responses.

**Python concepts:**
- `@property` decorator for read-only computed attributes.

---

## agent/tools/base.py + registry.py

**v0.1 keeps it ad-hoc.** We hardcode `{"add": AddTool()}` in `__main__`.
v0.5 promotes to a `ToolRegistry` with `pkgutil`-based auto-discovery.

**The tool contract being learned:** name, description, `schema()` returning
JSON schema for the LLM, `execute(**kwargs)` async returning the result.

---

## channels/base.py

**v0.1 keeps the CLI channel concrete, not ABC-based.** v0.11 introduces
`BaseChannel` ABC + `ChannelManager` when we add the WebSocket channel.

**The pattern to preserve so v0.11 is cheap:** the channel has `start()`,
methods that publish to bus, and a consumer task that reads from
`bus.consume_outbound`.
