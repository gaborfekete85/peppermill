# v0.3 Reading Notes — TurnState FSM + AgentRunner

> Notes from reading the nanobot originals before writing PepperMill v0.3.
> Sources: `../../../nanobot/nanobot/agent/loop.py` and `runner.py`.

## nanobot/agent/loop.py — TurnState

**The enum (`loop.py:63`):** 8 states — `RESTORE, COMPACT, COMMAND, BUILD, RUN, SAVE, RESPOND, DONE`.

**TurnContext (`loop.py:84`):** Mutable scratch space carried between states. Holds the inbound message, session reference, `history`, `initial_messages`, `final_content`, `tools_used`, `all_messages`, `stop_reason`, plus a bunch of callbacks (`on_progress`, `on_stream`, etc.) and observability metadata (state trace, latency). For v0.3 we copy only the strict minimum.

**Transition table (`loop.py:149`):** `_TRANSITIONS: dict[tuple[TurnState, str], TurnState]`. Handlers return event strings (`"ok"`, `"dispatch"`, `"shortcut"`); the driver looks up `(state, event) → next_state`. Sample rows:
```
(RESTORE, "ok")       → COMPACT
(COMPACT, "ok")       → COMMAND
(COMMAND, "dispatch") → BUILD
(COMMAND, "shortcut") → DONE
(BUILD, "ok")         → RUN
(RUN, "ok")           → SAVE
(SAVE, "ok")          → RESPOND
(RESPOND, "ok")       → DONE
```

The transition map is more flexible than direct returns when N-way branching happens (COMMAND splits into BUILD vs DONE), but for v0.3's purely linear 4-state FSM (BUILD → RUN → RESPOND → DONE) it's overkill. We use `match/case` and have handlers return the next state directly. If branching ever appears, switch to a transition map.

**State handlers (`loop.py:1275+`):** Each handler is `async def _state_X(self, ctx: TurnContext) -> str` returning an event string. They mutate `ctx` in place — that's how data flows between states.

## nanobot/agent/runner.py — AgentRunner

**Dataclass shapes:**
- `AgentRunSpec` (`runner.py:75`): `initial_messages`, tool registry, model, `max_iterations`, `max_tool_result_chars`, `concurrent_tools`, `context_window_tokens`, plus callbacks (`injection_callback`, `checkpoint_callback`). We copy only `initial_messages`, `tools`, `tool_schemas`, `provider`, `max_iterations`.
- `AgentRunResult` (`runner.py:105`): `final_content`, `messages`, `tools_used`, `usage`, `stop_reason`, `had_injections`. We copy `final_content`, `messages`, `tools_used`, `stop_reason`.

**The run loop (`runner.py:252`):** `for iteration in range(spec.max_iterations)`. Each iteration: prepare messages (context governance — we skip), call provider, check tool calls, execute, drain injections. If loop exhausts without a final text, `stop_reason = "max_iterations"`. v0.3 keeps the iteration cap but drops all the governance + injection logic.

**What v0.3 deliberately ignores:**
- `_drop_orphan_tool_results`, `_microcompact`, `_apply_tool_result_budget`, `_snip_history` — context governance (v0.8).
- `_drain_injections`, `injection_callback` — mid-turn message injection (v0.10).
- `checkpoint_callback`, runtime_checkpoint — crash recovery (v0.9).
- `_MAX_EMPTY_RETRIES`, `_MAX_LENGTH_RECOVERIES` — empty-response and length recovery.
- `chat_with_retry` — retries.
- `concurrent_tools`, `_partition_tool_batches` — tool concurrency (v0.10ish).

## Mapping nanobot → PepperMill v0.3

| nanobot | PepperMill v0.3 |
|---|---|
| `TurnState` (8 states) | `TurnState` (4 states: BUILD, RUN, RESPOND, DONE) |
| `_TRANSITIONS` dict + event strings | Handlers return `TurnState` directly + `match/case` routing |
| `TurnContext` (huge) | `_TurnContext` (small: inbound, messages, tool_schemas, result) |
| `AgentRunSpec` (~10 fields) | `AgentRunSpec` (5 fields) |
| `AgentRunResult` (~7 fields) | `AgentRunResult` (4 fields) |
| `_state_build` (history + memory + skills) | `_state_build` (just initial_messages + tool_schemas) |
| `_state_run` (full retry/inject/checkpoint machinery) | `_state_run` (build spec, call runner, store result) |
| `_state_respond` (assemble outbound, may suppress) | `_state_respond` (always publish outbound) |

## Python concepts to practice in v0.3

- **`enum.Enum` with `auto()`** — sentinel values for state names. Identity comparison: `state is TurnState.DONE`, not `==`.
- **`match/case`** — Python 3.10+ structural pattern matching. For enums, match on the value directly:
  ```python
  match state:
      case TurnState.BUILD:  ...
      case TurnState.RUN:    ...
  ```
  The `case _:` default catches "unknown state" — good defensive coding.
- **Mutable context dataclass** — pass one object between handlers; each handler mutates the fields it owns. Keeps handler signatures uniform (`(self, ctx) -> TurnState`) without globals.
- **Module separation by responsibility** — runner.py is "execute one LLM turn" (no bus, no channels). loop.py is "orchestrate one per-inbound turn". Each file holds together as a single concept.
- **Test doubles at different layers** — `ScriptedProvider` (existing) for runner tests, `ScriptedRunner` (new) for loop tests. Each test exercises one layer against the interface of the layer below.

## Open questions for later versions

- When does the transition map become worth introducing? Probably when COMMAND state arrives (v0.5+) with its two-way branch.
- How does `runtime_checkpoint` integration work (v0.9)? Will runner accept an `on_checkpoint` callback in its spec.
- Will the `_TurnContext` grow too large by v0.10? If so, split into smaller per-phase contexts.
